# Orchestrator Agent Permission Bootstrap Plan

> **Ticket:** TICKET-1520 (`7d0c8cf8`)  
> **Date:** 2026-02-20  
> **Status:** Planning

## Objective

Add a one-time installation process that pre-configures agent permissions for Orchestrator protocol access, so agents can call the local API without repeated approval prompts.

## Problem Summary

Current ticket workflows require the agent to call Orchestrator endpoints (`attach`, `update`, `deliver`, etc.). In restricted terminal environments, those calls often trigger approval prompts unless command prefixes are already trusted. This creates friction and can interrupt execution flow.

## Desired Outcome

After running a single installer command during setup, the agent can run all required Orchestrator protocol calls without further permission prompts, while keeping scope tightly limited to safe local API paths.

## Scope

- In scope:
  - Add an installer for permission bootstrap.
  - Define exact command/endpoint allowlist needed for protocol flow.
  - Add verification and troubleshooting steps.
  - Integrate the installer into local setup docs and launch flow guidance.
- Out of scope:
  - Redesigning the protocol API.
  - Broadly disabling sandbox/approval systems.
  - Production cloud auth changes.

## Required Permission Surface

Minimum protocol operations that must be pre-approved:

1. `POST /api/protocol/attach`
2. `POST /api/protocol/update`
3. `POST /api/protocol/decision`
4. `POST /api/protocol/ask`
5. `POST /api/protocol/read-context`
6. `POST /api/protocol/write-context`
7. `POST /api/protocol/deliver`
8. `GET /api/protocol/context/:ticketId` (launch-time prompt fetch)

## Proposed Implementation Plan

### Phase 0: Capability Spike (1 short task)

1. Confirm current writable config locations and schemas for each supported agent runtime:
   - Claude (`.claude/settings.local.json` allowlist merge)
   - Codex (`.codex/config.toml` and/or runtime-approved prefix persistence model)
2. Decide whether Codex permissions are file-configurable, approval-prefix based, or hybrid.
3. Lock a stable config contract before writing installer logic.

Deliverable: a short capability note committed to docs or shared context.

### Phase 1: Permission Installer Script

Add `scripts/install-agent-permissions.mjs` with:

1. `--agent=claude|codex|all` target selection.
2. `--platform-url` override (default `http://localhost:3000`).
3. Safe idempotent merge behavior:
   - Read config.
   - Add only missing allow entries.
   - Preserve existing user entries.
4. Automatic backup before write (timestamped copy).
5. `--dry-run` mode to preview changes.

If a runtime cannot be preconfigured by file, fallback behavior:

1. Print a deterministic “permission warmup” command set.
2. Run optional warmup sequence once so user can approve scoped prefixes and persist them.

### Phase 2: Verification Command

Add `scripts/verify-agent-permissions.mjs` and a package script:

1. Validate presence of all required allowlist entries.
2. Report missing permissions with exact remediation command.
3. Exit non-zero on failure (usable in setup automation/CI for local environments).

### Phase 3: Setup and UX Integration

1. Update `docs/mvp-local-setup.md`:
   - Add “Install agent permissions” immediately after token setup.
   - Add verify command and troubleshooting section.
2. Add launch-path guidance (ticket launch UI docs):
   - If permissions are missing, show a one-line warning with installer command.
3. Optional: add a lightweight runtime check in launch workflow to surface missing setup early.

### Phase 4: Validation and Rollout

1. Manual validation matrix:
   - Fresh machine (no existing configs)
   - Existing configs with custom entries
   - Custom non-local `OVERLORD_URL`
2. Confirm protocol runs end-to-end without prompts after install.
3. Document rollback:
   - Restore from backup file generated by installer.

## Security Guardrails

1. Scope command permissions to Orchestrator protocol endpoints only.
2. Default host scope to local Orchestrator URL (`localhost:3000`) unless user overrides.
3. Never persist `AGENT_TOKEN` in config files.
4. Avoid broad wildcards (`curl:*`) in generated entries.
5. Log exactly what the installer changed for auditability.

## Acceptance Criteria

1. One installer command prepares required agent permissions.
2. A verification command confirms readiness and lists any missing entries.
3. Running a full ticket flow (`attach` -> updates -> `deliver`) no longer triggers repetitive permission approval prompts.
4. Existing user permission config remains intact except for additive scoped entries.
5. Setup docs include install, verify, and rollback instructions.

## Risks and Mitigations

1. Agent config format drift:
   - Mitigation: keep parser/serializer minimal and validate format before write.
2. Over-permissive defaults:
   - Mitigation: endpoint-level allowlist and explicit host/path scope.
3. Cross-platform path differences:
   - Mitigation: resolve paths from project root and user home, with clear “not found” guidance.

## Recommended Next Steps

1. Implement Phase 0 capability spike first (small, low risk).
2. Build installer + verifier scripts.
3. Update docs and launch flow messaging in same PR.
4. Run manual validation on a clean environment before merging.

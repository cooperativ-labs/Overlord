# Security Audit - 2026-05-25

## Executive Summary

- Reviewed the Overlord repository security posture for the web app, protocol APIs, Supabase Edge Functions, database policies, storage policies, authentication flows, and deployment configuration.
- The core authenticated Next.js and protocol API surfaces show a generally acceptable access-control posture for customer evaluation: server routes authenticate callers, protocol routes validate organization membership before using service-role access, and Supabase RLS policies enforce organization and ticket-level boundaries.
- One issue should be remediated before broad customer reliance on hosted automation: multiple Supabase Edge Functions are callable without verified caller authentication while using the Supabase service-role key. This can allow unauthorized background mutations, notification abuse, AI-provider spend, and cross-tenant data-derived feed generation.
- No immediate customer-side action is required based on this code review. Remediation is primarily platform-side: add internal authentication to service-role Edge Functions, rate-limit unauthenticated device-code issuance, and strengthen browser defense-in-depth headers.

## Product Surface Reviews

### Edge Functions, Background Jobs, and Integrations

#### Finding: Service-role Edge Functions are publicly callable
- Severity: High
- Why it matters: Functions that bypass RLS with `SUPABASE_SERVICE_ROLE_KEY` must authenticate callers themselves. If deployed without Supabase JWT verification and without an internal trigger secret, an external caller can invoke privileged work directly. In this repository, that can trigger push notifications to organization members, refresh the agent model catalog, and generate or overwrite feed posts from ticket data.
- Evidence: `package.json` defines `functions-deploy` as `supabase functions deploy --project-ref zitmmhvbilhjjdwgxlfm --no-verify-jwt`. `supabase/config.toml` explicitly sets `verify_jwt = false` for `functions.generate-feed-post`. `supabase/functions/generate-feed-post/index.ts` accepts `ticketId`, `objectiveId`, and `organizationId` from the JSON body and uses a service-role client without checking an authorization header. `supabase/functions/send-push-notification/index.ts` sends notifications for a caller-provided `organizationId` using service-role reads and has no caller authentication. `supabase/functions/sync-agent-models/index.ts` mutates `agent_models` using the service-role key and has no caller authentication. By contrast, `send-newsletter`, `send-signup-notification`, and `generate-changelog-draft` implement explicit bearer or admin checks.
- Recommendation: Require a dedicated internal bearer secret or signed job token on every service-role Edge Function that is not intended for direct public use. Verify the token before parsing or processing the body. For `generate-feed-post`, derive `organization_id` from the ticket row rather than trusting caller input, and verify the ticket belongs to that organization before updating or inserting feed posts. Avoid deploying all functions with `--no-verify-jwt`; configure unauthenticated behavior per function.

### Authentication and Access Control

#### Finding: Device authorization request endpoint has no issuance rate limit
- Severity: Medium
- Why it matters: The device-code flow is intentionally unauthenticated at issuance time, but public creation without an IP, user-agent, or origin rate limit can be abused to fill `device_auth_codes`, drive database writes, and create approval-code noise. The polling route has a per-code throttle, but that does not limit creation of new codes.
- Evidence: `apps/web/app/api/auth/device/request/route.ts` accepts any `POST`, creates a UUID `device_code` and short `user_code`, and inserts into `device_auth_codes` with a service-role client. `apps/web/app/api/auth/device/poll/route.ts` throttles only by `next_poll_at` on an existing device code. `supabase/migrations/20260303000001_device-auth-poll-throttle.sql` adds poll throttling but no issuance throttle.
- Recommendation: Add rate limiting to `/api/auth/device/request`, preferably backed by a durable store keyed by IP and coarse user-agent, with conservative limits and short expiration. Add periodic cleanup for unapproved expired codes and monitor creation volume.

### Infrastructure and Deployment Configuration

#### Finding: Main application responses do not define a Content Security Policy
- Severity: Medium
- Why it matters: The app handles session-bearing pages, agent prompts, ticket content, markdown, OAuth flows, and local desktop integration. A CSP would materially reduce the impact of a future XSS or third-party script injection by restricting script, connect, frame, image, and form destinations.
- Evidence: `apps/web/next.config.ts` sets `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Strict-Transport-Security`, and `Permissions-Policy` globally. It defines `Content-Security-Policy` only for `/sw.js`, not for `/(.*)` application responses.
- Recommendation: Add a production CSP for all app responses. Start with `default-src 'self'`, explicit `script-src`, `connect-src` for Supabase, Sentry, Vercel, and required OAuth providers, `img-src` for configured image sources, `frame-ancestors 'none'`, and `base-uri 'self'`. Roll out in report-only mode first if needed.

### Database and Supabase Policies

#### Finding: Administrative authorization is tied to a hard-coded email address
- Severity: Low
- Why it matters: Hard-coded admin email checks are brittle for offboarding, account recovery, and delegation. If the mailbox remains active after role changes, administrative access remains available until code and database policies are updated.
- Evidence: `supabase/migrations/20260517110000_changelog_entries.sql` grants changelog draft management when `auth.jwt() ->> 'email' = 'jake@cooperativ.io'`. `supabase/migrations/20260520120000_mailing_list.sql` uses the same email check for mailing-list admin read access. `lib/actions/admin-features.ts` and `lib/actions/admin-agent-models.ts` use `isAdminEmail(user.email)` before service-role mutations.
- Recommendation: Move admin authorization into a database-backed role or membership model and reference that consistently from RLS and server actions. Keep an emergency break-glass path separate from everyday admin access.

## Positive Controls

- Protocol API routes consistently use `parseProtocolBody` or `resolveAgentToken` before creating service-role clients, and `resolveProtocolAuth` verifies Supabase JWTs against JWKS plus organization membership before allowing protocol work.
- Service-role client usage is centralized in `supabase/utils/service-role.ts` with a clear warning that callers must authenticate first.
- Ticket, objective, file-change, tag, execution-target, profile, push-token, and storage access policies use organization membership, role checks, or self-access constraints rather than broad authenticated access.
- Storage policies keep the `artifacts` bucket private and mirror ticket/organization access controls.
- A prior migration explicitly removed permissive invitation policies, and another revoked unnecessary `TRUNCATE`, `TRIGGER`, and `REFERENCES` grants from `anon` and `authenticated`.
- Auth flows use Supabase `auth.getUser()` for server-side user validation, and password sign-in/sign-up sanitize local `next` paths before redirecting.
- Security headers already include HSTS, frame denial, content-type sniffing protection, referrer policy, and restrictive permissions policy.

## Recommended Next Steps

- Add required internal authentication to `generate-feed-post`, `send-push-notification`, and `sync-agent-models`, then redeploy without relying on blanket `--no-verify-jwt`.
- Add tests or smoke checks that unauthenticated calls to privileged Edge Functions return `401`.
- Add issuance rate limiting and monitoring to `/api/auth/device/request`.
- Add an application-wide CSP in report-only mode, tune it against production telemetry, then enforce it.
- Replace hard-coded admin email checks with a database-backed admin role and update RLS policies/server actions to use that role.
